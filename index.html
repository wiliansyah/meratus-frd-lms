Meratus Academy LMSBab 1: Pendahuluan

1. PENDAHULUAN

1.1 Latar Belakang

Meratus Group, sebagai ekosistem maritim dan logistik terintegrasi terbesar di Indonesia, memiliki struktur organisasi berskala enterprise yang mencakup berbagai unit bisnis, multi-company, dan ratusan lokasi cabang (*site*). Dalam upaya menstandarisasi kompetensi ribuan karyawannya serta menjalankan kewajiban pelatihan K3 (HSE) dan kepatuhan (*Compliance*), inisiatif Meratus Academy dicanangkan.

Pengalaman menggunakan platform LMS sebelumnya memberikan banyak pelajaran penting (lessons learned) mengenai keterbatasan arsitektur pelaporan, kesulitan revisi konten, hingga kendala approval matrix yang tidak fleksibel. Oleh karenanya, dibutuhkan sebuah Learning Management System (LMS) generasi baru yang revolusioner. Sistem ini tidak sekadar menyimpan video/SCORM, melainkan juga harus mampu melakukan pemetaan kompetensi (*Skill Mapping*), analisis kesenjangan (*Gap Analysis*), penyusunan *Individual Development Plan* (IDP), serta terintegrasi langsung secara real-time/batch dengan sistem HRIS (Talenta) yang sudah ada.

1.2 Tujuan Dokumen

Dokumen Functional Requirements Document (FRD) ini merupakan landasan teknis (blueprint) yang disusun untuk:

  • Mendokumentasikan secara rinci ke-52 spesifikasi fungsional dan seluruh atribut non-fungsional (NFR) yang diwajibkan oleh Subject Matter Expert (Tim L&D Meratus).
  • Memetakan resolusi teknis atas pain points dari LMS yang sebelumnya digunakan agar tidak terulang pada pengembangan sistem baru.
  • Menjadi acuan tunggal dan final bagi tim Software Engineer, UI/UX Designer, dan Quality Assurance (QA) dalam membangun, menguji, dan meluncurkan sistem.

1.3 Ruang Lingkup Sistem

Pengembangan LMS Meratus Academy mencakup pembangunan aplikasi berbasis web dengan arsitektur SPA (Single Page Application) yang meliputi modul-modul berikut:

  • Modul Core & Governance: RBAC, Multi-Entity Setup, Branding, Master Data.
  • Modul Learner Experience: Katalog cerdas, E-Learning Player (SCORM, Video, PDF), Kuis, Profil Kompetensi, IDP.
  • Modul Trainer/Admin: Course Builder, Workflow Approval, Penjadwalan Multi-batch, Absensi, Penilaian *Assignment*.
  • Modul HRBP/Management: Compliance Dashboard, Evaluasi Kirkpatrick (Level 1-3), *Gap Analysis*, Custom Report Builder.

2. DESKRIPSI UMUM SISTEM

2.1 Perspektif Produk

LMS Meratus Academy adalah Custom Web Application yang berdiri secara terpusat. Aplikasi ini dibangun dari nol (ground-up) demi alasan keamanan dan fleksibilitas *Business Logic* yang kompleks. Sistem ini beroperasi di *Cloud Infrastructure*, memanfaatkan integrasi *Single Sign-On (SSO)* dari Azure AD / Google Workspace perusahaan untuk akses mulus (seamless).

2.2 Karakteristik Pengguna (Aktor)

Sistem ini dirancang untuk mengakomodasi berbagai level aktor dalam organisasi Meratus:

2.3 Asumsi dan Ketergantungan (Dependencies)

  • Data Source: API/SFTP dari sistem HRIS (Talenta/SAP) Meratus harus siap mengirimkan data karyawan harian secara andal untuk *batch sync*.
  • Otentikasi: Infrastruktur *Identity Provider* (Azure AD) Meratus telah dikonfigurasi untuk menerima *SAML assertion* atau *OAuth token* dari LMS.
  • Konten Awal: Tim L&D telah menyusun dan merapikan setidaknya 1 versi awal "Kamus Kompetensi" dalam format Excel sebelum implementasi sistem untuk di-*seed* ke dalam Master Data.

2.4 Batasan Sistem (System Constraints)

  • Sistem tidak bertanggung jawab atas proses perhitungan *Payroll* atau *Key Performance Indicator (KPI)* tahunan. Data pelatihan dari LMS dapat ditarik oleh sistem KPI eksternal via API, namun perhitungan insentif tidak dilakukan di LMS.
  • Akses sistem difokuskan melalui Web Browser Responsif (Mobile Web). Tidak ada pembuatan aplikasi *Native* (APK/IPA) terpisah pada fase 1 ini.
  • Pemutar Video internal tidak menggantikan fungsi *Video Conferencing*. Untuk kelas *Live*, sistem akan menautkan (*embed/redirect*) ke aplikasi *MS Teams / Zoom* yang telah berlisensi di Meratus.

System Notification
(Admin, Learner)		 Tidak diberi info terkait sistem update dan notifikasi pembaruan dan dampaknya apa, sehingga membuat user & admin kebingungan. Disrupsi operasional dan ketidaknyamanan pengguna akibat fitur yang tiba-tiba berubah tanpa panduan. 9. NON-FUNCTIONAL REQUIREMENTS (NFR)

Bab ini merumuskan batasan atribut kualitas, kinerja kuantitatif, metrik pengalaman pengguna, keamanan arsitektur, dan ketersediaan server. Penetapan standar di bawah ini ditujukan untuk memfasilitasi kebutuhan Enterprise Meratus Group dengan tingkat skalabilitas prima.

9.1 Performance & Scalability (Kinerja & Skalabilitas)NFR-01
Response Time Halaman Statis
Muat halaman (Page Load Time) di modul dashboard UI, katalog penelusuran, dan profil harus berada di < 3 Detik (pada profil jaringan normal 4G/LTE 10 Mbps).
API Response Time (p95)
Tanggapan pemrosesan logika REST API backend (kecuali operasi bulk report) harus konsisten di bawah < 500 Milidetik untuk persentil ke-95.
Media Streaming & Aset
Delivery video harus menggunakan protokol HLS (HTTP Live Streaming) atau Adaptive Bitrate via CDN (Content Delivery Network). Waktu Time-to-First-Frame saat menekan 'Play' harus < 3 Detik untuk mereduksi kebosanan buffering.
Concurrent Users Capacity
Infrastruktur web server, DB connection pooling, dan Redis Cache (opsional) harus dirancang agar stabil melayani > 500+ Pengguna Bersamaan (Concurrent) saat ada *blast* sesi Pelatihan Compliance Serentak (*Traffic Spike*).
9.2 Security, Privacy & Compliance (Keamanan Lapis Baja)NFR-02
Otentikasi Utama (SSO & MFA)
Wajib memblokir pintu masuk tradisional dan mengimplementasikan protokol SAML 2.0 / OpenID Connect / OAuth 2.0. Hal ini mendelegasikan keamanan (*termasuk MFA*) kepada Identity Provider (IdP) Meratus yakni Microsoft Azure AD / Google Workspace.
Session Management Timeout
Token kredensial (*JWT / Cookies HttpOnly*) diprogram agar *Expired / Ter-Logout Otomatis* apabila portal dibiarkan menganggur (Idle Time) melebihi batas ambang 8 Jam kerja bagi Learner.
Data Encryption & Transport
Injeksi trafik data mutlak diamankan oleh sertifikat digital HTTPS/TLS 1.2+ di semua endpoint. Password pengguna eksternal (akun vendor) di-hash minimum dengan standar Bcrypt (Salt Rounds ≥ 10). Tidak ada sandi berformat *Plain Text* di Database.
Vulnerability Prevention
Kerangka kerja kode pemrograman (Framework) harus mengimplementasikan teknik pembendungan *SQL Injection* (Via Prepared Statements/ORM), mitigasi *XSS* (Sanitasi Input Input Parser HTML), dan perlindungan Cross-Site Request Forgery (CSRF Tokens).
Data Privacy & Masking Setup
Sistem menyediakan lembar kesepakatan *Consent / T&C* saat awal orientasi Login. Mendukung Data Masking otomatis pada level UI: Nomor Identitas sensitif (Gaji, NIK Pribadi, Nomor Rekening jika ada) disensor (*asterisk*) terhadap instruktur non-karyawan atau admin berhak rendah untuk mematuhi regulasi UU PDP. Mekanisme Data Retention Policy aktif otomatis mendaur-ulang/membersihkan data PII pengguna yang telah pensiun lama.

11.2 Referensi Rekomendasi Tech Stack & Infrastruktur

Pemilihan tumpukan teknologi didasarkan pada komparasi performa, stabilitas komunitas Enterprise, dan kemudahan maintenance di tangan IT Meratus ke depan.

Layering KomponenTeknologi Rekomendasi (Alternatif)Keterangan Kapabilitas / Alasan
Frontend & SPA React.js (Next.js) / Vue.js + Tailwind CSS Reaktivitas halaman instan, tidak me-reload DOM saat navigasi modul SCORM. Optimalisasi komponen modular (*Netflix-style* UI).
Backend Core API Node.js (NestJS / Express) / Go-lang Sifat I/O *Non-blocking* sangat baik untuk *streaming* data ping SCORM masal dan melayani ribuan koneksi *concurrent* yang konstan.
Database (RDBMS) PostgreSQL (Atau setara MySQL) Kekuatan jaminan transaksi *ACID*. Handal memetakan relasi kompleks *Foreign Key* ganda antara NIK, Course, Entity, dan Hierarki Log Audit.
Object Storage / CDN AWS S3 / Google Cloud Storage + Cloudflare CDN Pembebanan file statis (Video MP4 besar, File Zip Ekstraksi SCORM, PDF Aset, Sertifikat) di ember terpisah di luar RAM Computing Server utama.
Web Server Proxy Nginx Membendung *Reverse proxy*, *Load balancing*, kompresi gzip/brotli, dan pemutusan persilangan aman protokol HTTPS *SSL Certificate (Let's Encrypt)*.
Auth & Cache SAML 2.0 (SSO Integration) + Redis (Opsional) Mendelegasikan sekuriti sandi dan token mutlak ke ranah MS Azure. Redis mempercepat (*caching*) hasil hitung *Leaderboard* laporan.
